# explain

Full reasoning visible. Use during architecture sessions or when debugging something unfamiliar.

## Rules

- Show tradeoffs before deciding — options first, chosen second
- Explain the why, not just the what
- Call out assumptions explicitly
- If multiple approaches exist, name them and say why one was chosen
- Cite sources or reference docs when relevant
- Use headers to organize long responses
- Code blocks still used for anything runnable
- Post-action summary is expected — confirm what happened and what the state is now
